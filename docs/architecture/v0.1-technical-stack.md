# V0.1 Technical Stack

## Status

Accepted

## Purpose

This document records the accepted v0.1 technical architecture and stack for Djoko Studio. It documents the decisions already made by the project owner and architecture mentor, without introducing new stack choices.

## Architecture Shape

Djoko Studio v0.1 is a polyglot monorepo with a service-oriented architecture.

The repository is organized around a small set of focused services rather than a single monolithic application. Different languages are used where they best match the runtime and operational needs of each service.

### Target Monorepo Structure

```txt
apps/
  web-studio/          TypeScript + Vite + React

services/
  api/                 Go REST/gRPC API: users, studios, sessions, auth
  signaling/           Go WebSocket signaling: rooms, offers, answers, ICE
  upload/              Rust service: tus/chunks/checksum/resume
  media-worker/        Rust worker: assemble, validate, call FFmpeg
  export-worker/       Rust worker: render final 1080p via FFmpeg

packages/
  contracts/           OpenAPI/Protobuf schemas
  shared-types/        Generated types for frontend/backend

infra/
  docker-compose.yml
  monitoring/
  local object storage
  local event bus
  local TURN/STUN service

docs/
  adr/
  architecture/
  product/
```

## Selected Stack

Frontend:
React + TypeScript + Vite

Browser recording:
MediaRecorder for v0.1
WebCodecs later

Browser local persistence:
OPFS for media chunks
IndexedDB for metadata/manifests/upload state

Live call:
WebRTC P2P for v0.1
coturn for TURN/STUN relay

Signaling:
Go WebSocket service

Core API:
Go service

Upload:
Rust + Axum/Tokio
tus resumable upload protocol

Database:
PostgreSQL

Object storage:
S3-compatible API
MinIO for local/dev
S3/R2-compatible provider later in production

Events/queue:
NATS JetStream

Media processing:
Rust workers orchestrating FFmpeg

Rendering:
FFmpeg CPU rendering for MVP

Infrastructure:
Docker
Docker Compose

## Why This Shape

- TypeScript is used for the browser application where rich UI, state management, and shared contracts matter most.
- Go is used for network-oriented services such as API and signaling because those services are primarily about request handling, session state, and transport coordination.
- Rust is used for performance-sensitive upload and media processing services where reliability, concurrency, and control over I/O matter most.
- FFmpeg is used for media rendering because it is the accepted rendering backbone for the MVP export pipeline.
- PostgreSQL stores relational metadata for studios, users, sessions, uploads, and processing state.
- S3-compatible object storage stores media files, chunks, raw tracks, and exports.
- NATS JetStream coordinates async media jobs and events across the pipeline.
- Docker Compose runs the local and development environment.

## Explicit Non-Decision

Reverse proxy / gateway:
Deferred decision.
No Nginx, Caddy, or Traefik choice is made yet.
Local/dev uses Docker Compose directly.
Production routing, TLS, WebSocket proxying and upload gateway concerns will be decided later through a separate ADR.
